# PRODUCTION PLAN — Escrowa (proof-of-production)

## Live URL
- **Board + demo:** Vercel — `escrowa.edycu.dev` (Next.js).
- **Escrowa service:** Fly.io / Railway — holds the `T3nClient` TEE session + escrow key custody.

## Settlement proof (the headline artifact)
- A release fires in the demo via the **simulated** `signing` + `outbox` host, producing a settlement reference on screen. The Rust→WASM contract logic and the secp256k1 signatures are **real**; the settlement reference is generated by the simulated host (not an on-chain tx — T3N is production-ready for when the network launches, see the "Hackathon Simulation Context" note in README).
- Escrowa `did:t3n` + party DIDs registered via the simulated **`did-registry` / `agent-registry`** (`board/src/sdk/didRegistry.ts`) — include the DIDs.

## Published package
- Publish the escrow client as npm **`escrowa-sdk`** — `createMilestone()`, `attest()`, `release()`. A judge can fund a milestone and watch a release fire.

## Tests
- **54 Vitest test cases** (15 app/API + 39 SDK; 100% coverage; 115+ assertions incl. a 100-case matrix) + **18 Rust contract tests** = **72 total**.
- Coverage: the 4 release paths (mutual / deadline / arbiter / unfunded-reject), signature verification against did:t3n, **agent-auth scope enforcement** (`host/agent.function_denied` + `host/http.egress_denied`), and a **key-custody test that fails if the escrow private key ever appears in logs, env, or disk**.

## Benchmark
- `scripts/bench.py` → p50/p95/mean over 200 release-condition evaluations (excluding chain settlement). Methodology noted.

## Verify / integrity
- `scripts/verify_release.ts` — replays `m1` and asserts a settlement reference is produced and the key never exported.
- `scripts/verify_no_unilateral.ts` — asserts no single-party attestation can trigger release.

## Settlement reference
- Release settlement reference (from the simulated `signing` + `outbox` host) + `did-registry` DIDs — in README. Labeled as simulated, not an on-chain/explorer link.

## Readiness gate
- `scripts/check_submission_readiness.py` scans the shipped docs (`README.md`, `docs/*.md`, `board/`/`contract/` READMEs) for leftover angle-bracket placeholders (repo, vercel-url, link, explorer-link, ref) **and** asserts the mandatory deliverables exist (`BUGS.md`, `docs/DEMO.md`, `docs/ARCHITECTURE.md`, `scripts/bench.py`).
